## 关键设计原则

1. **宁可少做，不要做残** — MVP 只做 B站/知乎/纯文本三个来源，其他 v2
2. **数据不丢** — 后台任务执行失败后可恢复，不因重启丢失
3. **不安全内容不进前端** — 所有抓取内容清洗后再展示
4. **离线可用** — 纯本地部署，不依赖外部服务（LLM 除外）

## 分层说明

### 1. 前端层 (Frontend)
- **框架**: Next.js App Router + TypeScript
- **样式**: Tailwind CSS
- **状态**: TanStack Query (React Query) 管理服务端状态
- **BFF 模式**: Next.js Route Handler 作为代理层，前端不直接调 FastAPI
- **页面路由**:
  - `/` — 仪表盘
  - `/ingest` — 提交链接/上传文件
  - `/tags` — 标签列表
  - `/tags/[id]` — 标签详情 + 聚合总结
  - `/links/[id]` — 链接详情
  - `/query` — 问答界面
  - `/search` — 搜索

### 2. API 路由层 (Routers)
- 职责：参数校验、权限控制、请求/响应序列化
- 不包含业务逻辑
- 统一错误格式

### 3. 服务层 (Services)
- 所有业务逻辑集中在这里
- **Crawler**: 多平台内容抓取（策略模式）
- **Processor**: AI 处理管线（摘要 + 打标签 + 向量嵌入）
- **TaskRunner**: 后台任务调度 + 失败重试 + 重启恢复
- **RAG**: 检索增强问答
- **Dedup**: 基于 URL + 内容哈希的去重检测

### 4. 数据层 (Data)
- **SQLite + SQLAlchemy**: 结构化数据存储
- **ChromaDB**: 向量数据库，用于语义搜索
- 两部分通过 link_id 关联

## 异步处理管线

### 架构

```
发链接 → 创建 Task 记录 (SQLite) → 返回 202 → TaskRunner 轮询执行
                           ↑                              │
                           └────── 重启后恢复未完成任务 ───┘
```

### Task 数据模型 (links 表 status 字段)

每条 Link 自带 status 字段，充当任务状态机：

```
pending ──→ processing ──→ completed
                │
                └──→ failed (记录 error)
```

任务执行核心逻辑：

```python
class TaskRunner:
    """后台任务调度器，启动时自动恢复未完成任务"""

    async def start(self):
        """应用启动时，将所有 status=pending 的链接重新提交处理"""
        unfinished = await db.execute(
            select(Link).where(Link.status.in_(["pending", "processing"]))
        )
        for link in unfinished:
            asyncio.create_task(self._run_pipeline(link.id))

    async def _run_pipeline(self, link_id: int):
        """单条内容处理管线，含重试和错误处理"""
        try:
            link = await db.get(Link, link_id)
            link.status = "processing"
            await db.commit()
            # ... 执行抓取 + AI 处理
            link.status = "completed"
        except Exception as e:
            link.status = "failed"
            link.error = str(e)
            await db.commit()

    async def retry_failed(self, link_id: int):
        """手动重试失败的任务"""
        link = await db.get(Link, link_id)
        link.status = "pending"
        link.error = None
        await db.commit()
        asyncio.create_task(self._run_pipeline(link_id))
```

### 关键设计
- **重启不丢任务**：启动时扫描所有 pending/processing 的链接，重新执行
- **重试策略**：LLM 调用失败自动重试 2 次（指数退避 1s → 3s）；抓取失败不自动重试（大概率是链接问题）
- **手动重试**：前端提供"重新处理"按钮

## 安全设计

### XSS 防护

Crawler 抓取的内容（尤其知乎/通用 fallback 的 HTML）必须清洗后才能存储和展示：

```python
# services/cleaner.py
import re

def sanitize_content(html: str) -> str:
    """清洗 HTML，只保留纯文本，去掉所有标签"""
    # 1. 移除 script/style 标签及其内容
    html = re.sub(r'<script[^>]*>.*?</script>', '', html, flags=re.DOTALL)
    html = re.sub(r'<style[^>]*>.*?</style>', '', html, flags=re.DOTALL)
    # 2. 移除其他所有 HTML 标签
    text = re.sub(r'<[^>]+>', '', html)
    # 3. 解码 HTML 实体
    text = html.unescape(text)
    return text.strip()
```

前端展示时使用 `{textContent}` 而非 `innerHTML`，React 默认会做转义。

### 内容安全

- 所有爬取内容统一经过 `sanitize_content()` 再存入 `content` 字段
- `raw_content` 保留原始 HTML，仅供调试查看，前端不展示
- 用户输入的文本（纯文本提交）不做清洗，但不允许 HTML

## LLM 调用策略

```python
async def call_llm_with_retry(llm_func, max_retries=2):
    """带指数退避的 LLM 调用"""
    for attempt in range(max_retries + 1):
        try:
            return await llm_func()
        except (RateLimitError, APITimeoutError) as e:
            if attempt == max_retries:
                raise LLMError(f"LLM 调用失败: {e}")
            wait = (2 ** attempt)  # 1s → 3s
            await asyncio.sleep(wait)
```

| 策略 | 值 |
|------|-----|
| 最大重试次数 | 2 |
| 退避策略 | 指数: 1s → 3s |
| 不重试的异常 | 401(API key 无效)、400(请求格式错误) |
| API Key 配置 | 通过 `.env` 的 `LLM_API_KEY` |
