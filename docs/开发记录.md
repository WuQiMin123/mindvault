# 开发记录

## 阶段回顾

### Phase 0: 项目脚手架

初始化项目结构：FastAPI 后端骨架、pyproject.toml 依赖管理、Dockerfile + docker-compose、CI (GitHub Actions)、ruff 配置。

**决策：** 直接上 async（FastAPI + SQLAlchemy async），因为后续的 API 调用和抓取都是 IO 密集型。同步改异步是 painful 的迁移，一开始就做对更省事。

### Phase 1: 后端核心 API

完成链接/标签的 CRUD、ILIKE 全文搜索、SQLite 作为初始数据库。

**决策：** FTS5 全文搜索 / ILIKE 就够了。知识库的量级不会大到需要向量检索，而且向量检索引入的复杂度（embedding 服务、向量数据库、召回评测）对 solo 项目来说太重了。ILIKE 在大几百条数据量级下体验完全可以接受。

### Phase 2: 内容摄入 + AI 管线

B站字幕 API 对接、知乎 Playwright 抓取、LLM 摘要/打标签管线。引入 BackgroundTasks 做异步处理。

**遇到的问题：**

- **知乎反爬严格**：Playwright 容易被检测，headless 模式下成功率不稳定。最终定位是用浏览器插件让用户手动选中内容替代全自动抓取，插件也在 Phase 5 实现了。
- **B站字幕 API 需要 cookie**：直接请求 player/v2 接口返回空数据，加上 credentials: 'include' 后才正常工作。

### Phase 3: Web 前端

Next.js App Router + Tailwind CSS + shadcn/ui。仪表盘、提交页、标签浏览、聚合总结、搜索。

这个阶段比较顺利，没有遇到特别棘手的 bug。

### Phase 5: 浏览器插件

Manifest V3 插件，支持右键菜单保存选中文本、快捷键 Ctrl+Shift+V、B站字幕自动提取。

**遇到的核心问题：**

- **chrome.storage.session 在 content script 中不可用**：这是 Chrome MV3 的限制。content script 不能读写 session storage。解决方式是通过 `chrome.runtime.sendMessage` 把数据发给 background service worker，由 background 代为存储。
- **B站 __INITIAL_STATE__ 获取**：content script 运行在 isolated world，无法直接访问页面的 `window.__INITIAL_STATE__`。最初用 script 注入 + postMessage 方式，但太脆弱（页面结构变化就失效）。最终改为从 URL 提取 bvid，通过 B站公开 API 获取视频信息，完全不依赖页面上下文。
- **host_permissions 缺失**：Manifest V3 中 popup 的 fetch 请求需要显式声明 `host_permissions`，否则报 "Failed to fetch"。

### Phase 7: 收尾

修复 Dockerfile 构建路径、补充多阶段构建、README、seed 演示数据、Makefile 扩充。

## 沉淀的方法论

### 一、异常处理原则

**不捕不知道该怎么处理的异常。** 如果 catch 块只是 `pass` 或 `print`，说明调用方不需要处理它，那就让异常往上抛，直到一个知道怎么处理的地方。代码中的每一个 try/catch 都应该有明确的含义：这是边界（用户输入校验）、这是重试时机（网络超时）、这是降级（LLM 不可用时跳过 AI 处理）。

### 二、配置优先于代码

API key、模型名、数据库 URL 这些应该永远在配置里，不在代码里。`llm_api_key` 默认值设为空字符串而不是占位符——空 key 意味着"未配置"，代码可以据此做合理降级（跳过 AI 处理）而不是带着假 key 去撞墙。`.env.example` 作为模板提交，真正的 `.env` 在 `.gitignore` 里。

**教训：** 缺少 `.env` 文件导致 AI 摘要静默失败，查了一个小时才定位到是 key 为空。如果加个启动时检查配置的 log，这个 bug 可以在 5 分钟内发现。

### 三、响应的正确性优先于性能

PATCH 端点的 bug（#2、#4）都源于同一个模式：先构建响应，再修改数据。当响应已经被序列化或部分构建后，后续的数据变更不会反映在返回值里。如果 API 返回了不正确的数据，性能再快也没有意义。

**规则：** 响应应该在所有数据变更完成后构建，或者在变更后显式更新响应中的相关字段。

### 四、浏览器插件开发的特殊约束

Manifest V3 引入了几条重要的限制，跟 Manifest V2 的思路完全不同：

1. **Service Worker 取代 Background Page** — 不能长期持有状态，需要靠 storage 或消息传递做持久化。
2. **Content Script 权限受限** — `chrome.storage.session` 不可用，需要用 `sendMessage` 中转。
3. **Isolated World 隔离** — content script 无法直接访问页面 JS 变量，有几种解法：
   - **推荐：** 直接从 URL / DOM / API 获取所需信息，不依赖页面 JS 上下文
   - 次选：注入 `<script>` 标签 + postMessage（需要处理消息时序和 SPA 导航）
   - 避免：从 `innerHTML` 正则解析（脆弱、性能差）
4. **host_permissions 显式声明** — 任何跨域请求都需要在 manifest 中声明，否则静默失败。

### 五、技术选型取舍

| 场景 | 选择 | 不选 | 理由 |
|------|------|------|------|
| 搜索 | ILIKE / FTS5 | 向量检索 | 数据量级小，向量检索引入的复杂度（embedding 服务、向量数据库、召回评测）不值得 |
| 数据库 | SQLite → PostgreSQL 迁移计划 | 一开始就用 PostgreSQL | SQLite 零配置启动，对 solo 项目友好。多用户是未来需求，不提前优化 |
| 前端框架 | Next.js App Router | Vite / CRA | App Router 的布局嵌套、服务端组件、流式渲染在知识库场景中都能发挥作用 |
| AI | OpenAI 兼容 API | 本地模型 | 本地模型（ollama 等）的响应质量和速度差距明显，API 调用在 solo 项目中成本可控 |
| 内容提取 | 用户主动选择（浏览器插件） | 全自动抓取 | 反爬成本越来越高，手动选择反而更精准——用户只存自己关心的内容 |

### 六、开发节奏

**先做可行性验证，再做详细设计，然后编码。**

Phase 2 开始前先 spike 了各平台的抓取可行性（spike/ 目录下的测试脚本），发现 B站有字幕 API 可以直接用、知乎需要 Playwright 且不稳定、抖音几乎不可行。这些验证结果直接影响了技术方案——B站走 API、知乎降低优先级、抖音放弃。如果先写代码再验证，可能会在不可行的方案上浪费大量时间。

**一次只做一个 phase，做完再回头优化。**

每个 phase 都有明确的边界和交付物。Phase 1 不写前端，Phase 2 不优化搜索。这避免了"顺便修一下"带来的 scope creep。

### 七、文档策略

- **架构决策记录**（docs/ 下的 .md 文件）：记录 WHY，让未来的自己能理解当时为什么做某个选择。
- **API 接口设计**：前后端协作的契约，先定接口再实现。
- **README**：给新用户看的，只写"怎么跑起来"和"有什么功能"。
- **不写注释，写代码。** 好的代码应该自我表达。注释只解释 WHY（为什么这样写），不解释 WHAT（在做什么）。

### 八、工具使用

- **任务列表** 用于多步骤任务跟踪，避免遗漏。
- **Plan mode** 用于架构决策前对齐，减少返工。
- **专注的工具**：Read 读文件，Grep 搜代码，Write/Edit 写文件，Bash 只做 shell-only 操作。每个工具都有明确的职责边界。
