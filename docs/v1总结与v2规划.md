# MindVault v1 — 总结与 v2 规划

---

## 第一部分：v1 版本总结

### 项目概况

MindVault 是一个人知识库系统，核心流程：收集内容 → AI 自动处理 → 标签归档 → 搜索检索。

v1 完成了从零到可用的全栈建设，涵盖后端 API、Web 前端、浏览器插件三大端，所有代码已推送至 GitHub（`WuQiMin123/mindvault`）。

### 项目规模

| 度量 | 数值 |
|------|------|
| 提交文件数 | 116 个 |
| 总代码行数 | ~18,000 行 |
| Python 源文件 | 32 个 |
| TypeScript/React 文件 | 21 个 |
| API 端点 | 17 个 |
| 测试用例 | 6 个 |
| HTML/CSS/配置等 | ~60 个 |

### 技术栈（实际落地）

| 层 | 技术 |
|------|------|
| 后端框架 | FastAPI (async) + SQLAlchemy (async) + aiosqlite |
| 数据库 | SQLite (WAL 模式) |
| Web 前端 | Next.js 16 (App Router) + React 19 + Tailwind CSS 4 + shadcn/ui |
| 浏览器插件 | Manifest V3 (Chrome/Edge) |
| AI | OpenAI 兼容 API (DeepSeek V4 Flash) |
| 内容抓取 | httpx + B站官方 API + Playwright (知乎) |
| 工程 | Docker + docker-compose + pytest + ruff + GitHub Actions |

### 功能清单

#### 后端 (17 个 API 端点)

| 分组 | 端点 | 说明 |
|------|------|------|
| **内容摄入** | `POST /ingest/link` | 提交链接，后台自动抓取 + AI 处理 |
| | `POST /ingest/bookmark` | 快速暂存链接，归入"待处理"标签 |
| | `POST /ingest/text` | 保存纯文本/笔记，支持标签 |
| | `POST /ingest/check` | 检查 URL 是否已存在（去重） |
| **内容管理** | `GET /links` | 内容列表（分页、按标签筛选） |
| | `GET /links/{id}` | 内容详情 |
| | `PATCH /links/{id}` | 修改内容 + 手动触发"未成功"重试 |
| | `PATCH /links/{id}/read-status` | 切换已读/未读状态 |
| | `DELETE /links/{id}` | 删除内容 |
| | `POST /links/{id}/analyze` | 手动触发 AI 重新处理 |
| **标签管理** | `GET /tags` | 标签列表（含链接数） |
| | `POST /tags` | 创建标签 |
| | `PUT /tags/{id}` | 修改标签名称/颜色 |
| | `DELETE /tags/{id}` | 删除标签 |
| | `PUT /tags/links/{link_id}/tags` | 批量设链接标签 |
| | `GET /tags/{id}/summary` | 标签聚合总结（AI 生成） |
| **搜索** | `GET /search?q=&tag=` | 关键词搜索 + 标签筛选 |

#### Web 前端 (7 个页面)

| 路由 | 功能 |
|------|------|
| `/` | 仪表盘：今日统计、摘要动态、最近内容列表 |
| `/ingest` | 提交链接或纯文本 |
| `/tags` | 标签列表（可创建/编辑/删除） |
| `/tags/{id}` | 标签内内容列表 + 聚合总结 |
| `/links/{id}` | 内容详情（编辑、标签管理、重分析） |
| `/search` | 关键词 + 标签组合搜索 |
| `/login` / `/register` | 登录注册（预留，当前未启用认证） |

#### 浏览器插件 (Manifest V3)

- **右键菜单**：选中文本 → 右键"保存到 MindVault"
- **快捷键**：`Ctrl+Shift+V` 一键保存选中内容
- **B站字幕提取**：在 B站视频页面点击插件图标 → 自动获取 CC 字幕 → 保存
- **弹出浮窗**：可编辑内容、添加标签、配置 API 地址

### 数据处理管线

```
提交链接 → status=pending → 返回 202
    │
    ▼
TaskRunner 轮询执行:
    1. Crawler 抓取（B站 API / 知乎 Playwright / 通用 httpx）
    2. 内容清洗（sanitize_content 去 HTML 标签）
    3. LLM 摘要生成（DeepSeek V4 Flash）
    4. LLM 自动打标签（1-3 个）
    5. 失败处理 → status=no_content，归入"未成功"标签
    6. 完成 → status=completed
```

### 关键架构决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 同步/异步 | 全 async | IO 密集型场景（API 调用、抓取）async 优势明显，sync 改 async 迁移成本高 |
| 数据库 | SQLite (WAL) | 零配置，solo 项目足够。PostgreSQL 留给多用户阶段 |
| 搜索 | ILIKE / 全文搜索 | 无需向量检索复杂度，几百条数据 ILIKE 体验足够 |
| 内容提取 | 用户主动选择 > 自动抓取 | 反爬成本日益增高，浏览器插件让用户手动选中反而更精准 |
| AI 调用 | 开源 API > 本地模型 | 本地模型响应质量和速度差距明显，API 成本对 solo 项目可控 |
| 前端 | Next.js App Router | 布局嵌套、服务端组件、流式渲染在知识库场景中天然适用 |

### v1 Bug 复盘

共 6 个 bug，按根因分类：

- **低级错误（4 个）**：import 遗漏、SQLAlchemy async 使用方法不当、缺少配置文件、manifest 权限声明遗漏
- **平台约束（2 个）**：`chrome.storage.session` 在 content script 中不可用、content script 无法访问页面 JS 上下文

**教训：** 4 个低级错误中有 2 个可以通过启动时日志检查发现（配置缺失、权限不足），1 个是类型检查盲区（动态返回类型未 import）。引入更好的启动时自检可以显著减少这类问题。

### 方法论沉淀

详见 `docs/开发记录.md`，核心原则：

1. **异常处理**：不捕不知道该怎么处理的异常
2. **配置优先**：API key 等配置不在代码里，空 key 做合理降级
3. **响应正确性 > 性能**：数据全部变更完成后再构建响应
4. **MV3 约束**：content script 优先从 URL/API 获取信息，不依赖页面 JS 上下文
5. **先验证再编码**：spike 验证可行性后再做详细设计
6. **一次一个 phase**：明确边界，避免 scope creep

## 第二部分：v1.1 规划

### v1.1 定位

在 v1 单用户基础上，**优先打磨日常使用体验，再做移动端扩展**。多用户和 PostgreSQL 属于过早优化，现阶段不做。

核心思路：**先用好 → 再移动 → 再多人**。这是个人知识库，先让自己用得舒服更重要。

### Phase 顺序

| Phase | 内容 | 预估 |
|-------|------|------|
| **P0** | 数据导出 + 前端状态补齐 | 2 天 |
| **P1** | 移动端 Web 响应式优化 | 1 天 |
| **P2** | 移动端 App (Expo) | 2 周 |
| **P3** | 批量操作 + 测试覆盖 | 3 天 |

### P0: 体验完善（基础）

**目标：** 补齐 v1 缺失的体验细节，让日常使用更顺手。

**数据导出：**

用户对自己的数据有完全控制权，支持一键导出：

```
格式:
  JSON — 完整数据备份（links + tags，不含向量）
  Markdown — 可读性好，每篇一个 .md 文件，ZIP 打包

API:
  GET /api/v1/export/json
  GET /api/v1/export/markdown
```

**前端状态补齐：**

所有页面补齐三种状态，解决"提交后不知道成功没有"、"列表到底有没有数据"等体验问题：

| 组件 | loading | empty | error |
|------|---------|-------|-------|
| 仪表盘 | Skeleton loading | 引导去提交 | 重试提示 |
| 内容列表 | Skeleton loading | "暂无内容" + 引导 | 错误提示 + 重试 |
| 标签页 | Skeleton loading | "创建第一个标签" | 错误提示 |
| 搜索页 | 搜索中 spinner | "未找到相关内容" | 错误提示 |
| 提交页 | 提交按钮禁用 + spinner | — | 错误提示条 |

**启动时配置自检：**

在 `lifespan` 中检查关键配置，提前暴露问题：

```python
# app/main.py lifespan 内
if not settings.llm_api_key:
    logger.warning("LLM_API_KEY 未配置 — AI 摘要和打标签功能将不可用")
```

### P1: 移动端 Web 响应式

**目标：** 移动端浏览器可用，零成本覆盖手机使用场景。在原生 App 之前先优化 Web 移动端体验。

**改动范围：**

```
frontend/src/
├── app/
│   ├── layout.tsx     — 导航栏改为底部 tab bar（移动端） / 侧边栏（桌面端）
│   ├── page.tsx       — 卡片改为单列布局
│   ├── ingest/        — 全屏输入，提交按钮固定底部
│   ├── search/        — 搜索框吸顶，结果列表自适应
│   └── tags/          — 网格改为列表
└── components/
    └── nav-bar.tsx    — 响应式导航（抽屉或底部导航）
```

- 使用 Tailwind 的 `sm:` / `md:` breakpoint 做布局切换
- 触摸友好：增大点击区域（min 44px）
- 不引入额外依赖

### P3: 效率 + 质量

**目标：** 日常操作效率提升 + 测试覆盖提供重构安全感。

**批量操作：**

| 操作 | 场景 |

| 批量加标签 | 多篇内容归入同一主题 |
| 批量标记已读 | 一次性清掉阅读队列 |

前端实现：列表进入选择模式 → 勾选 → 顶部操作栏出现。

### 不做的事（明确排除 v1.1）

- **多用户 / JWT 认证** — 个人知识库场景，现阶段不需要
- **PostgreSQL 迁移** — SQLite + WAL 对 solo 项目足够健壮，无并发写入问题
- **向量检索 / RAG** — v1 已决策，数据量级不到（排除理由不变）
- **离线模式** — 以服务端为唯一真实数据源（排除理由不变）
- **抖音/公众号抓取** — v1 spike 已验证抖音不可行，公众号无公开 API，均排除
- **WebSocket 实时推送** — 非实时场景，轮询足够
- **Admin 后台** — 无管理后台需求
