# 技术可行性评估

> 对每个关键技术选型评估风险、给出评级和替代方案。

---

## 综合结论

```
可行性: ████████░░ 80%  →  大部分选型低风险，3 个中风险需关注
风险点: 知乎反爬 / B站无字幕 / LLM 成本
```

---

## 逐项评估

### 1. Backend: FastAPI + SQLAlchemy (async)

| 维度 | 评估 |
|------|------|
| 可行性 | ✅ 高 |
| 风险 | 低 |
| 说明 | FastAPI async + aiosqlite 是成熟组合。SQLAlchemy 2.0 async 支持稳定 |
| 注意 | 必须用 WAL 模式 (`?journal_mode=WAL`)，否则写入冲突会让异步请求互相阻塞 |

### 2. B站内容获取 (字幕 API)

| 维度 | 评估 |
|------|------|
| 可行性 | ⚠️ 中高 |
| 风险 | 中 |
| 路径 | `view` → `player/v2` → subtitle URL → JSON 字幕 → 拼接文本 |
| 不稳定的点 | **大量视频没有字幕**（尤其个人UP主、游戏实况、直播切片）。根据经验，约 40% 的 B站视频没有 AI 字幕，即使有也有 10-20% 的识别错误率 |
| 缓解方案 | 如果没有字幕，走通用 Fallback（视频标题 + 简介 + 评论提取？），或前端提示"该视频暂无字幕" |

### 3. 知乎内容获取 (页面解析)

| 维度 | 评估 |
|------|------|
| 可行性 | ⚠️ 中 |
| 风险 | 中高 |
| 路径 | httpx GET → BeautifulSoup 解析 `.RichText` 节点 |
| 不稳定的点 | **知乎反爬严格**。未登录状态下：1) 部分文章只显示摘要 2) 频繁请求会被 ban IP 3) HTML 结构变动时解析会断 |
| 缓解方案 | 1) 加 User-Agent 轮换 2) 请求间隔 1s 以上 3) 如果失败走通用 Fallback（用 readability 库提取） |

### 4. 通用网页 Fallback

| 维度 | 评估 |
|------|------|
| 可行性 | ✅ 高 |
| 风险 | 低 |
| 路径 | httpx + BeautifulSoup + meta 标签 (`og:title`, `og:description`) |
| 说明 | 不需要精确解析，能拿到标题 + 正文片段就算成功。适合外链类内容 |

### 5. LLM 云端 API (DeepSeek)

| 维度 | 评估 |
|------|------|
| 可行性 | ✅ 高 |
| 风险 | 中 |
| 路径 | OpenAI 兼容 API: `chat/completions` + `embeddings` |
| 成本估算 | DeepSeek 定价: 输入 ¥0.5/M tokens, 输出 ¥2/M tokens。每篇文章摘要+标签+向量大约消耗 2-5K tokens。按每天存 10 条算，月成本约 **¥3-5**，可以接受 |
| 注意 | **embedding 模型必须跟 Chat 模型分开配置**。DeepSeek 的 embedding 能力不如 OpenAI `text-embedding-3-small`。建议 Chat 用 DeepSeek，Embedding 用 OpenAI（后者有免费额度） |

### 6. ChromaDB

| 维度 | 评估 |
|------|------|
| 可行性 | ✅ 高 |
| 风险 | 低 |
| 路径 | `PersistentClient(path="./data/chroma")` |
| 注意 | ChromaDB 0.5+ 在 Windows 上 hnswlib 偶有兼容问题，建议固定版本。只用 cosine 距离，不用其他 metric |
| 数据安全 | 向量不计入备份（JSON 导出不含向量），因为可以从 content 重建 |

### 7. TaskRunner (后台任务 + 重启恢复)

| 维度 | 评估 |
|------|------|
| 可行性 | ✅ 高 |
| 风险 | 低 |
| 说明 | 不需要 Celery/Redis。单用户场景，`asyncio.create_task` 就够用 |
| 注意 | `create_task` 创建的任务不会被 `asyncio.run()` 等待，启动时务必确保 lifespan 中的启动逻辑跑完再接受请求 |

### 8. Next.js + TanStack Query

| 维度 | 评估 |
|------|------|
| 可行性 | ✅ 高 |
| 风险 | 低 |
| 说明 | 标准组合。BFF 模式用 `next.config.ts` 的 `rewrites` 代理到 FastAPI |
| 注意 | D3.js 不需要（本项目没有图谱），避免引入不必要的依赖 |

### 9. Docker 部署

| 维度 | 评估 |
|------|------|
| 可行性 | ✅ 高 |
| 风险 | 低 |
| 路径 | `docker compose up -d` 包含 backend + frontend |
| 注意 | SQLite 文件放到 volume 里持久化，避免容器重建导致数据丢失 |

## 风险评估矩阵

```
高风险 (立即处理)     中风险 (持续关注)      低风险 (接受)
──────────────────────────────────────────────────────────
知乎反爬被 ban        B站无字幕视频         SQLite WAL 兼容性
                       LLM API 成本         ChromaDB hnswlib 兼容
                       通用 Fallback 效果差   DeepSeek embedding 弱
                       服务重启时序问题
```

## Windows 环境特别注意事项

| 问题 | 影响 | 处理方式 |
|------|------|----------|
| `aiosqlite` Windows 兼容 | 无影响，纯 Python | 正常使用 |
| ChromaDB hnswlib | 偶见 DLL 加载失败 | 固定 chromadb==0.5.5 版本 |
| Docker Desktop 资源占用 | 开发体验变差 | 开发期可以直接本机跑，不用 Docker |
| 文件名编码 (中文) | PDF 上传可能乱码 | 上传时统一重命名为 UUID |

## 环境依赖

```
后端依赖 (轻量):
  fastapi, uvicorn, sqlalchemy, aiosqlite,
  pydantic-settings, httpx, beautifulsoup4,
  openai, chromadb, structlog

前端依赖 (标准):
  next, react, tailwindcss, @tanstack/react-query

系统工具:
  Python 3.11+, Node.js 22+, Docker (可选)
```

整个项目 **不依赖任何需要编译的 C/C++ 扩展库**（除了 ChromaDB 的 hnswlib，但 pip 会自动处理）。
