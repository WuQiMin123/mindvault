# 数据模型设计

## ER 图概览

```
Link ──< LinkTag >── Tag
```

## Link（链接/内容）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | 自增主键 |
| url | TEXT | 来源链接 (可选，纯文本无链接) |
| title | TEXT | 标题 |
| content | TEXT | 清洗后的正文 (可为空，提取失败时仅保留链接) |
| raw_content | TEXT | 原始内容 (抓取原文，可为空) |
| source | TEXT | 来源平台: bilibili / zhihu / wechat / douyin / pdf / manual / extension |
| summary | TEXT | AI 生成的摘要 (可为空) |
| tags_ai | TEXT | AI 建议的标签 (JSON 数组，如 ["AI","编程"]) |
| content_hash | TEXT | 内容哈希 (MD5, 用于去重检测) |
| is_read | INTEGER | 阅读状态: 0=未读, 1=已读 |
| status | TEXT | pending / processing / completed / failed / no_content |
| error | TEXT | 失败原因 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |

### 说明
- `tags_ai` 是 AI 建议的标签，用户确认后写入 link_tag 关联表
- `content` 是清洗后的可读文本（**用于语义搜索和 RAG 问答**），提取失败时可为空
- `raw_content` 保留原始抓取结果（用于排查问题），提取失败时可为空
- `summary` 是 AI 生成的摘要（**用于列表预览和标签聚合总结**），可为空
- **完整内容（content）和摘要（summary）各自独立**：全文供搜索/问答，摘要供快速浏览。手动录入时两者都写，链接提取失败时两者都为空
- `status` 跟踪异步处理管线状态:
  - `pending` → 已入库等待处理
  - `processing` → 正在抓取/AI 处理中
  - `completed` → 全部处理完成（含内容提取成功）
  - `no_content` → 处理完成但内容提取失败，仅保留标题+链接
  - `failed` → 处理过程发生异常（非预期错误）
- `content_hash` 是内容正文的 MD5，提交新链接时先查 hash 避免重复入库
- `is_read` 用于阅读队列管理，用户可标记"稍后读"或"已读"

## Tag（标签）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | 自增主键 |
| name | TEXT UNIQUE | 标签名 |
| color | TEXT | 显示颜色 (hex) |
| created_at | DATETIME | 创建时间 |

### 说明
- 标签全局唯一，不允许重名
- 颜色前端随机生成，可用户修改
- 标签由 AI 自动建议或用户手动创建
- 系统自动管理的标签（如"未成功"、"待处理"）由后端在首次使用时自动创建，用户可手动删除或改名

## LinkTag（链接-标签 多对多关联）

| 字段 | 类型 | 说明 |
|------|------|------|
| link_id | INTEGER FK | 关联 Link.id |
| tag_id | INTEGER FK | 关联 Tag.id |

### 联合主键 (link_id, tag_id)

## 设计原则
- **SQLite 存储所有数据**：链接、标签、关联关系、全文索引 (FTS5)
- **不存大文件**：PDF 解析后只保留文本，原始 PDF 不在 DB 中
- **不存音视频**：B站/抖音只保留转录后的文本
- **基于哈希去重**：入库前计算 content_hash，已存在的内容直接返回旧记录
- **导出即资产**：支持一键导出全部数据（JSON/Markdown），用户对自己的数据有完全控制权
